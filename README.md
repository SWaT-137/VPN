# 🛡️ Geocint_VPN: Архитектурная документация проекта

Данный проект представлен в 4 версиях, 3 из которых базируются на едином высокопроизводительном ядре. Первая версия представляет собой связку таких протоколов как XRay, VLESS, Reality обеспечивающее максимальную защиту при правильной настройке. Вторая версия ("Призрак") направлена на обход глубокого анализа пакетов (Deep Packet Inspection) государственного уровня с использованием протокола Trojan. Третья версия ("Control Center") превращает проект в полноценную бекенд-систему с веб-интерфейсом администратора. Четвертая версия ("Radmin Style") является пользовательским десктопным приложением с графическим интерфейсом.

## 1. Единое ядро системы (Shared Core)
Ядро проекта спроектировано по принципу модульного "бутерброда", где каждый слой отвечает строго за свою задачу и не смешивается с другими. Это обеспечивает независимую масштабируемость и тестируемость.

### 1.1. Транспортный слой (Anti-DPI & Steganography)
* **Протокол:** Адаптация Trojan-протокола поверх TLS 1.2/1.3.
* **Обфускация:** Имитация обращения к популярному CDN (`speed.cloudflare.com`) с отсутствием ALPN, что заставляет DPI-системы классифицировать трафик как "неизвестное легитимное приложение".
* **Защита от статистического анализа:** Внедрен механизм динамического паддинга (0-31 байт случайного мусора на каждый пакет). Строгая структурная разделенность: `[Длина фрейма][Длина полезной нагрузки][Полезная нагрузка][Паддинг]`. Это исключает возможность определения размера реального IP-пакета.

### 1.2. Криптографический слой (End-to-End Encryption)
* **Алгоритм:** AES-256-GCM с рандомным 12-байтным вектором инициализации (Nonce) на каждый пакет.
* **Архитектура "Слепой почтальон" (Blind Router):** Сервер выступает только в роли маршрутизатора заголовков. Полезная нагрузка локальных пакетов (от клиента к клиенту) зашифрована ключом сети, который сервер не использует для дешифровки. Даже при полном компрометации сервера трафик внутри виртуальной локалки остается недоступным.
* **Флагирование маршрутизации:** Использование флагов (`0x00` для интернета, `0x01` для локалки) позволяет корректно разделять трафик, идущий на NAT (интернет), и трафик, требующий энд-ту-энд шифрования.

### 1.3. Слой аутентификации (Zero-Knowledge Lite)
* **Токенная система:** Каждый клиент идентифицируется с помощью уникального Pre-Shared Key (токена). На стороне клиента токен хэшируется (SHA-256) и по сети передается только хэш.
* **Управление доступом:** Администратор может отзывать доступ (ревокация) для конкретных узлов без изменения инфраструктуры всей сети.

### 1.4. Системный слой (Event-Driven I/O)
* **Драйвер:** Использование Wintun (Kernel-mode драйвер для Windows) через прямую обертку на `ctypes`.
* **Оптимизация ожидания:** Вместо синхронного `time.sleep()` или циклического `select()`, используется нативный вызов Windows API `WaitForSingleObject` на событиях (`ReadWaitEvent`) сессии Wintun. Ядро ОС само будит поток Python только при физическом поступлении пакета в интерфейс, что снижает задержку (Latency) до <1мс и загрузку CPU до 0.01%.

## 2. Ветвления проекта (Версии)

### Версия 1: XRay + Reality (Базовая реализация)
* **Статус:** Разработан.
* **Описание:** Использование технологической связки XRay + Reality, на 2026 год очень прочная связка для обхода государственных ограничений на иностранные и не только сервисы
* **Управление:** Через графический интерфейс

### Версия 2: "Ghost" (Инфраструктурный стелс)
* **Статус:** 📋 В разработке (Планирование).
* **Цель:** Обход DPI-систем активного зондирования (Active Probing) и SNI-фильтрации на уровне провайдеров (РКН, Great Firewall).
* **Ключевые изменения:**
  * Отказ от самоподписанных сертификатов в пользу легитимных Let's Encrypt.
  * Интеграция с Nginx (SNI Routing) для маскировки VPN-трафика под обычный HTTPS-сайт-приманку.
  * Генерация HTTP-шума (имитация структуры веб-трафика) внутри TLS-туннеля для обхода энтропийного анализа.

### Версия 3: "Control Center" (Бекенд и Мониторинг)
* **Статус:** 📋 В разработке (Планирование).
* **Цель:** Превращение сервера в управляемую микросервисную систему.
* **Ключевые изменения:**
  * Миграция хранилища с JSON-файлов на реляционную базу данных (SQLite) для обеспечения потокобезопасности.
  * Внедрение REST API на базе фреймворка FastAPI для удаленного управления.
  * Разработка Web-дашборда для мониторинга онлайн-статуса клиентов, сбора статистики трафика (RX/TX) и генерации пригласительных ссылок.

### Версия 4: "Radmin Style" (Десктопный клиент)
* **Статус:** 📋 В разработке (Планирование).
* **Цель:** Создание готового к дистрибуции коммерческого продукта для конечных пользователей.
* **Ключевые изменения:**
  * Разработка графического интерфейса на базе PyQt6 (темная тема, системный трей).
  * Асинхронное взаимодействие GUI с сетевым ядром.
  * Сборка всего проекта в единый исполняемый файл (`.exe`) с помощью PyInstaller без необходимости установки зависимостей на клиентских машинах.
